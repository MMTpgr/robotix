Cas d'utilisation : S'inscrire comme utilisateur.
But : Une personne souhaite s’inscrire en tant qu’utilisateur de robotix.
Préconditions : Le client fournit son nom, son prénom, son pseudo (qui doit être unique), son adresse courriel et son numéro de téléphone ainsi que le nom de la compagnie pour mieux le rediriger.
Acteurs : Client (principal), Système (secondaire).
Scénario principal : description étape par étape, ordonnée numériquement
1. Le client se rend sur l’interface du logiciel pour s’inscrire
2. l’Agent lui demande ses informations (nom,tel, etc…)
3. Le client entre ses données dans le système.
4. Le système valide les informations
5. Le système exige que le client lise et signe les conditions d’utilisations
6. Le système confirme la création du compte auprès du client
   6.1 Le client peut entrer en tant qu’utilisateur.
Scénarios alternatifs :
1a. Certaines informations sont jugées incorrectes.    1a.1 .Le système envoie un message d’erreur    1a.2. Le scénario reprend à l'étape 3. 2a. Les informations sont correctes mais déjà utilisées par un autre utilisateur    2.a.1 Le système envoie un message d’erreur    2.a.2 Le scénario reprend à l'étape 3. 3a. Le client refuse de signer les conditions d'utilisations    3.a.1 Le scénario se termine. Postconditions : Le système traite les besoins de l’utilisateur tant qu’il respecte les conditions d’utilisations.

Cas d'utilisation : Enregistrer un robot.
But : mise en contexte, description brève, préconditions nécessaires.
Préconditions : Le client a un compte robotix, possède le numéro de série, le nom et le type du robot.
Acteurs : Utilisateur (principal), Fournisseur (secondaire).
Scénario principal : 1. Le client se connecte à son compte
2. Le client demande le numéro de série auprès du fournisseur
3. Le client entre ce numéro dans le système ainsi que le nom et le type du robot qui identifient le robot.
4. Le système valide les informations
5. Le système confirme l’enregistrement du robot auprès du client
   5.1 Le client peut gérer son robot et accéder aux fonctionnalités affiliées.
Scénarios alternatifs : 1a. L’utilisateur se trompe dans l'entrée du numéro de série du robot.    1a.1. Le système envoie un message d’erreur    1a.2. Le scénario reprend à l'étape 3. 2a. Le nom du robot et le type du robot (qui servent d’identifiant) sont déjà utilisés.    2a.1. Le système envoie un message d’erreur    2a.2. Le scénario reprend à l'étape 3. 3a. Le client n’a pas de compte Robotix    3a.a Le scénario se termine Postconditions : Le système effectue toute les opérations demandées par l’utilisateur en lien avec le robot.

Cas d'utilisation : Acheter des composantes.
But : Un utilisateur souhaite acheter les composantes d’un robot .
Préconditions : Le client doit avoir un compte, l’argent nécessaire pour acheter une composante et doit acheter au moins un CPU et une autre composante quelconque.
Acteurs : Utilisateur (principal), un ou plusieurs fournisseurs (secondaire).
Scénario principal : 1. Le client se connecte à son compte 2. Le client contacte le fournisseur    2.1 l’utilisateur accède aux offres de composantes 3. Le client achète les composantes qu’il a choisit . 4. Le système valide l’achat et envoie le reçu à l'utilisateur 5. Le système sauvegarde les nouvelles composantes Scénarios alternatifs : 1a. L’utilisateur n’a pas assez d’argent pour acheter une composante.    1a.1 .Le fournisseur envoie un message d’erreur    1a.2. Le scénario reprend à l'étape 2.1. 2a. Le choix des composantes de l’utilisateur est invalide    2.a.1 CPU n’a pas été choisit ou le choix n’est pas assez varié    2.a.2 Le scénario reprend à l'étape 2.1. Postconditions : Le système a sauvegardé les nouvelles composantes sur le compte de l’utilisateur.

4. Risques (5%)

Identifiez 5 risques potentiels du projet et classez-les par ORDRE DE SEVERITE (voir la page 28 dans les diapositives des exigences (Chapitre 7) des notes de cours et page 90 du livre).

  1. Ne pas répondre correctement aux exigences: risque que le logiciel ne réponde pas à 100% aux exigences des clients à une estimation non précise de celles-ci. Par exemple, il se peut que la manière dont a été implémentée la gestion des tâches ne permet pas d’assigner directement les tâches aux robots.
  2. mauvaise structure générale: La structure générale du logiciel peut ne pas être robuste ce qui entraîne immédiatement beaucoup de problèmes si jamais on souhaite ajouter de nouvelles fonctionnalités au logiciel. Par exemple, si nous avons codé toutes les tâches de Robotix (mais suivant une faible architecture) et qu’on souhaite ajouter la gestion des tâches, il faudra redésigner toute la structure du logiciel ce qui peut être très indésirable.
  3. risques techniques: isque en lien avec la présence de code de moindre qualité qui comporte des erreurs de logique ou qui est souvent sujet à des bugs. Par exemple, le logiciel peut planter lors de gestion de robot en raison d’un bug/erreur de logique. Le logiciel peut aussi prendre beaucoup trop de temps pour réaliser une tâche dû à un code de mauvaise qualité ou encore montrer les mauvaises informations concernant le fournisseur ou les composantes.
  4. problèmes d'organisation: - problèmes de budget, problèmes en lien avec le respect des deadlines ou encore des problèmes de communication entre les développeurs qui peuvent retarder le projet et éventuellement mener à un mauvais respect des exigences et a du code de moindre qualité.
  5. Risques externes: risques peu probables d'événements externes tels que des phénomènes physiques qui peuvent retarder le développement. par exemple des catastrophes naturelles ou des affaires personnelles de certains développeurs qui ralentissaient.
